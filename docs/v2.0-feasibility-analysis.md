# 量化+AI 融合交易系统 v2.0 — 可行性分析

> 基于当前代码库 (commit 5f67a52) 逐项评估

---

## 总体评估

方案整体方向正确，大部分模块可在现有架构上增量实现。但有几项存在较高技术风险或成本收益比不佳，需要调整优先级或方案。下面逐项分析。

---

## 2.1 多智能体辩论机制 (Bull/Bear/Judge)

### 可行性: ✅ 可行，但需注意成本与延迟

**现有基础**:
- `AgentManager` 已支持多 Agent 创建/管理，quant-analyst Agent 已存在
- `AIRuntime.execute()` 支持 `agentId` 参数，可指定不同 Agent 执行
- 每个 Agent 有独立的 `IDENTITY.md` (系统提示词) 和工具权限配置
- Sub-agent 机制已实现 (`subAgentTools.ts`)，支持嵌套调用

**实现路径**:
1. 通过 `AgentManager.createAgent()` 创建 bull-analyst、bear-analyst、judge-analyst 三个 Agent
2. 在 `AutoTradingPipeline.processSignal()` 中，信号通过 SignalEvaluator 后，串行调用三个 Agent
3. Judge Agent 汇总多空意见，输出结构化 JSON (confidence + action)

**风险与约束**:
- **延迟**: 三次 LLM 调用串行执行，每次 5-15 秒，总延迟 15-45 秒/信号。对于日线级 Swing Trading 可接受，但不适合日内
- **成本**: 每个信号消耗 3 次 LLM 调用。若每日 50 个候选信号，按 DeepSeek-R1 约 $0.02/次计算，日成本约 $3；若用 GPT-4o 则约 $15/日
- **Token 消耗优化**: 方案中提到"将 UniverseScreener 范围缩小到仅筛选技术面多头排列的股票"是正确的，建议 Quant 预筛后仅 10-20 只进入 AI 辩论
- **结构化输出**: 需要 Judge Agent 严格输出 JSON，可通过 IDENTITY.md 中的 few-shot 示例约束，但 LLM 偶尔会输出非法格式，需要 fallback 解析

**建议调整**: 先实现 2-Agent 版本 (Analyst + Devil's Advocate)，验证效果后再加 Judge。减少一次 LLM 调用可降低 33% 延迟和成本。

## 2.2 ATR 动态仓位与 Chandelier Exit

### 可行性: ✅ 完全可行，改动量小

**现有基础**:
- `stock_analysis.py` 已计算 `ATR(14)` 并输出为 `atr14` 字段
- `StrategyEngine.scanWithStrategy()` 已将 `atr14` 纳入 indicator_values
- `StopLossManager` 已支持 `trailing_percent` 移动止损
- `QuantityCalculator` 已有三种模式框架，新增模式只需加一个 case
- `ohlcv_daily` 表已缓存历史数据，可直接计算 ATR

**实现路径**:

1. **QuantityCalculator 新增 `volatility_parity` 模式**:
```typescript
case 'volatility_parity': {
  const maxRiskAmount = (params.total_assets ?? 0) * 0.01;
  const riskPerShare = 2 * (params.atr14 ?? params.entry_price * 0.02);
  result = Math.floor(maxRiskAmount / riskPerShare);
  break;
}
```
改动: `QuantityCalculator.ts` 加一个 case + `types.ts` 的 `QuantityParams` 加 `atr14` 字段。

2. **StopLossManager Chandelier Exit**: 现有 `trailing_percent` 是百分比模式。需新增 `trailing_atr_multiplier` 字段，止损位 = `highest_price - multiplier * ATR`。改动: `StopLossRecord` 类型加字段 + `checkAll()` 中加 ATR 分支 + schema 迁移加列。

3. **ATR 数据传递**: `AutoTradingPipeline.processSignal()` 需在下单前获取 ATR。可从 `ohlcv_daily` 表直接计算，或从 `stock_analysis.py` 的输出中读取。

**风险**: 几乎无风险。ATR 数据已存在，仅需管道串联。

## 2.3 智能拆单 (TWAP/VWAP) 与条件单

### 可行性: ⚠️ 部分可行，TWAP 简单，VWAP 复杂

**TWAP (时间加权拆单)**:
- 实现简单: 在 `TradingGateway.placeOrder()` 中，当 `orderAmount > threshold` 时，拆成 N 笔子订单，用 `setInterval` 间隔执行
- 改动量: TradingGateway 加 ~50 行拆单逻辑
- 风险: 低。但需处理部分子订单失败的回滚逻辑

**VWAP (成交量加权拆单)**:
- 需要日内分钟级成交量分布数据来计算权重
- 免费 Longport 账户无法获取日内分钟级数据
- **结论: 不可行**，除非升级行情套餐

**Stop-Limit 条件单**:
- `LongportAdapter` 已支持 `stop` 和 `stop_limit` 订单类型映射 (MIT/LIT)
- `PaperTradingEngine` 目前对 stop/stop_limit 仅返回 `submitted` 不模拟触发
- 改动: PaperTradingEngine 需加条件单触发逻辑 (在 `checkPendingOrders` 中比较价格)
- 风险: 低，但 PaperTradingEngine 的条件单模拟需要定时喂价格

**建议**: Sprint 2 先做 TWAP + Stop-Limit，VWAP 留到有实时分钟数据后再做。

## 2.4 实时行情升级

### 可行性: ⚠️ 技术可行，但受限于外部服务

**现状分析**:
- `QuoteService` 已是事件驱动架构 (`EventEmitter`)，支持 `on('price', callback)`
- 已有 WebSocket 推送 (`setOnQuote`) + 轮询双模式
- 免费 Longport 账户 US 行情延迟 ~15 分钟

**升级选项**:

| 方案 | 成本 | 延迟 | 改动量 |
|------|------|------|--------|
| Longport Level 1 实时 | ~$30/月 | 实时 | 零改动 (同一 SDK) |
| Alpaca Market Data | 免费 (基础) | 实时 | 新增 AlpacaQuoteProvider |
| Polygon.io Basic | $29/月 | 实时 | 新增 PolygonQuoteProvider |
| Finnhub WebSocket | 免费 | 实时 (有限) | 新增 FinnhubQuoteProvider |

**推荐方案**: Longport Level 1 升级，零代码改动，仅需更换账户套餐。若预算有限，Alpaca 免费实时行情是最佳替代，但需新增一个 QuoteProvider 适配器 (~200 行)。

**StopLossManager 事件驱动改造**:
方案中提到的 `StopLossManager.onTick(data)` 模式，当前 `checkAll()` 是定时轮询所有记录。改为事件驱动需要:
1. `QuoteService.on('price', (data) => stopLossManager.checkSymbol(data.symbol, data.lastPrice))`
2. 新增 `checkSymbol()` 方法，仅检查该 symbol 的活跃记录
3. 改动量约 30 行，风险低

**RealtimeEngine (日内分钟级指标)**:
- 需要在内存中维护分钟级 K 线并实时计算 MACD/VWAP
- 这是一个全新模块 (~300-500 行)
- 对 Swing Trading (持仓数天) 来说优先级不高，建议 P2

## 3.1 平仓后自动 PnL 归因

### 可行性: ✅ 完全可行，改动量极小

**现有基础**:
- `TradingGateway.recordStrategyUsage()` 已在成交时调用 `StrategyAllocator.recordUsage()`
- `StrategyAllocator.recordPnl()` 方法已实现
- 缺失的是: 卖出成交时未调用 `recordPnl()`

**实现**:
在 `TradingGateway.recordStrategyUsage()` 中，当 `order.side === 'sell'` 时:
```typescript
if (order.side === 'sell' && order.filled_quantity && order.filled_price) {
  // 查找原始买入均价计算 PnL
  const positions = await this.getPositions();
  const pos = positions.find(p => p.symbol === order.symbol);
  const avgCost = pos?.avg_cost ?? order.filled_price;
  const pnl = (order.filled_price - avgCost) * order.filled_quantity;
  const released = order.filled_quantity * order.filled_price;
  this.strategyAllocator.recordPnl(order.strategy_id, pnl, released);
}
```

改动: ~10 行。风险: 极低。

### AI 周末复盘 (Review Agent)

**可行性**: ✅ 可行。通过 `CronScheduler.registerHandler('weekly-review', ...)` + `AIRuntime.execute()` 实现。将亏损交易日志拼成 prompt 发给 AI，输出存入新表或推送通知。改动量约 50-80 行。

## 3.2 实时 VIX 接入

### 可行性: ✅ 可行

**实现路径**:
- yfinance 支持 `^VIX` 实时查询: `yf.Ticker("^VIX").info['regularMarketPrice']`
- 新增 Python 脚本 `scripts/vix_monitor.py`，输出当前 VIX 值
- 注册 cron handler `vix-update`，每 15 分钟执行一次
- 调用 `RiskController.updateDynamicRisk(portfolioDrawdown, vixLevel)`

**VIX 联动收紧存量止损**:
方案中提到"VIX > 25 时收紧所有 Trailing Stop 从 2*ATR 到 1*ATR"。这需要:
1. `StopLossManager` 新增 `tightenAllStops(multiplier)` 方法
2. 遍历所有活跃记录，重新计算 stop_loss
3. 改动量约 20 行，但需注意: 收紧后不应再放松 (单向操作)

**风险**: yfinance 的 `^VIX` 数据也有延迟 (~15 分钟)，但对于 VIX 级别的判断 (>25 vs <15) 足够。

## 3.3 BacktestEngine

### 可行性: ✅ 可行，且基础已存在

**现有基础**:
- `scripts/backtest_engine.py` 已实现完整回测逻辑 (含指标计算、信号生成、交易模拟、指标输出)
- `StrategyEngine.runBacktest()` 已封装 Python 调用并存储结果到 `backtest_results` 表
- `StrategyEngine.runParameterOptimization()` 已实现网格搜索优化
- `backtest_results` 表已建好

**方案中提到"重用 PaperTradingEngine 跑历史数据"**:
这个思路有问题。PaperTradingEngine 是 TypeScript 同步引擎，将历史 OHLCV 伪造成 tick 喂入 AutoTradingPipeline 会非常慢 (涉及 DB 读写、LLM 调用等)。

**更好的方案**: 继续使用现有的 Python `backtest_engine.py`，它已经能:
- 读取 yfinance 历史数据
- 计算技术指标
- 模拟交易 (含手续费和滑点)
- 输出 Sharpe/回撤/胜率等指标

需要补充的是:
1. 前端 `PerformanceView.tsx` 中展示回测结果 (已有 `backtest_results` 表)
2. 回测结果对比功能 (多策略/多参数)
3. 权益曲线可视化 (ECharts)

改动量: 前端 ~200 行，后端几乎不需要改。

## 关键风险总结

| 风险 | 等级 | 说明 | 缓解措施 |
|------|------|------|----------|
| LLM 结构化输出不稳定 | 🟡 中 | Judge Agent 可能输出非法 JSON | 加 JSON schema 校验 + fallback 解析 |
| 实时行情成本 | 🟡 中 | 免费方案有限制 | 先用 Longport L1 ($30/月) 或 Alpaca 免费 |
| VWAP 拆单不可行 | 🔴 高 | 缺少分钟级成交量数据 | 改用 TWAP，VWAP 留到有数据后 |
| 多 Agent 延迟 | 🟡 中 | 3 次 LLM 调用 15-45 秒 | Swing Trading 可接受；可并行 Bull/Bear |
| BacktestEngine 方案偏差 | 🟢 低 | 方案建议重用 PaperTradingEngine，实际 Python 引擎更优 | 继续用现有 backtest_engine.py |
| ATR 数据可用性 | 🟢 低 | stock_analysis.py 已计算 ATR | 无风险 |
